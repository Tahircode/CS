# Part 1: What is a Shallow Copy?
- A shallow copy is a bit-by-bit copy of an object. When the compiler generates a default copy constructor, it performs a shallow copy.

- This means it simply copies the value of every member variable from the source object to the destination object.

- For primitive members like int or double, this is fine. The value 5 is copied.
- For a pointer member, this is a disaster. The address stored in the pointer is copied, not the data that the pointer points to.

#include <iostream>

class ResourceHolder {
private:
    int* data;      // Our pointer member
    int size;       // A regular member

public:
    // Parameterized Constructor
    ResourceHolder(int s) : size(s) {
        data = new int[size]; // Allocate memory on the heap
        for (int i = 0; i < size; ++i) {
            data[i] = i * 10; // Fill it with some data
        }
    }

    // Destructor
    ~ResourceHolder() {
        
        delete[] data; // Free the heap memory
        data = nullptr; // Good practice
    }

    void printAddress() {
        std::cout << "Object's data pointer points to address: " << data << std::endl;
    }
};

int main() {
    // --- Step A: Create the first object ---

    ResourceHolder obj1(5); // Calls constructor, allocates memory
    obj1.printAddress();

    // --- Step B: Create the copy ---

    ResourceHolder obj2 = obj1; // Default copy constructor does a SHALLOW COPY
    obj2.printAddress();

    // --- Step C: Delete the first object ---
    // obj1 goes out of scope here, its destructor is called.
    // The memory at the address stored in obj1.data is now FREED.
    
    // --- Step D: Delete the second object ---
    // obj2 goes out of scope here, its destructor is called.
    // It will try to free the SAME memory that obj1 just freed.
    
    return 0;
}


**Memory State at Each Step**
1. Step A: After ResourceHolder obj1(5);
2. Step B: After ResourceHolder obj2 = obj1; (The Shallow Copy)
The default copy constructor copies the size (5) and the value of the data pointer (0x100).

Both obj1.data and obj2.data now point to the exact same block of memory!

3. Step C: After obj1 is destroyed
obj1's destructor runs. It calls delete[] data;. The memory at 0x100 is returned to the system. It is now invalid.

obj2 still exists, but its data member is now a dangling pointer—it points to memory that is no longer valid. Using this pointer would be "undefined behavior."

4. Step D: When we try to delete obj2
obj2's destructor runs. It calls delete[] data;. It tries to free the memory at 0x100 a second time.

**The Final Result - The Double Free Error**
What happens is a Double Free Error.

- The program's memory manager keeps track of which blocks of memory are allocated and which are free.
- When obj1's destructor runs, it tells the memory manager, "The block at 0x100 is now free."
- When obj2's destructor runs, it tells the memory manager, "The block at 0x100 is now free."
- The memory manager gets confused. It was already told that block is free. Trying to free it again can corrupt its internal state, leading to a crash. The program will almost always terminate abnormally.
