## Access Modifiers 
- At its core, an access modifier is a keyword in a programming language that specifies the accessibility or visibility of a class, its methods, its fields (attributes), and its properties.

- Types:
 . private
 . default / (Package-Private)
 . protected
 . public

 1. **private**
 - The Black Box
 - Core Idea: The most restrictive level. A private member (field or method) is accessible only from within the same class where it is declared. Nothing outside the class—not even a subclass—can see it or touch it.
 - How It Works: When you declare a field as private, you are signaling that this data is critical to the object's internal integrity. No other class should be able to manipulate it directly. Instead, you provide public methods (like "getters" and "setters") to control access.
  
  // BankAccount.java
public class BankAccount {
    // This field is PRIVATE. It cannot be accessed directly from outside this class.
    private double balance;
    private String accountHolderName;

    public BankAccount(String name, double initialDeposit) {
        this.accountHolderName = name;
        // We can access 'balance' here because we are inside the BankAccount class.
        this.balance = initialDeposit; 
    }

    // This is a PUBLIC "getter" method. It provides read-only access to the balance.
    public double getBalance() {
        return this.balance;
    }

    // This is a PUBLIC method that controls HOW the balance is changed.
    // It contains business logic to protect the object's state.
    public void deposit(double amount) {
        if (amount > 0) {
            this.balance += amount;
            System.out.println("Deposit successful. New balance: " + this.balance);
        } else {
            System.out.println("Invalid deposit amount.");
        }
    }
}

// Main.java
public class Main {
    public static void main(String[] args) {
        BankAccount myAccount = new BankAccount("Alice", 100);
        
        // CORRECT: Using the public method to interact with the object.
        myAccount.deposit(50); 
        System.out.println("Current Balance: " + myAccount.getBalance());

        // COMPILE ERROR! 'balance' has private access in BankAccount
        // myAccount.balance = 1000000; // <-- This line will not compile!
    }
}
**use-cases**

- All internal fields (attributes): Almost all of an object's data should be private by default. This is the single most important best practice in OOP.
- Internal helper methods: If you have a method inside your class that is only used by other methods within that same class (e.g., a validateEmail() method called by a setEmailAddress() method), make it private. It's an implementation detail that the outside world doesn't need to know about.

2. **default (Package-Private)**

- The Neighborhood Secret
- Core Idea: This level has no keyword. If you don't specify an access modifier, the member is default. A default member is accessible only to classes within the same package. It's invisible to the outside world and to subclasses in different packages.
- How It Works: This is a middle ground. It's less restrictive than private but more restrictive than protected. It's useful for creating tightly-coupled classes that work together as a module but need to hide their internal cooperation from the rest of the application.

// File: com/mybank/core/BankAccount.java
package com.mybank.core;

public class BankAccount {
    // No access modifier means 'default' (package-private)
    String accountNumber; // Accessible within the 'com.mybank.core' package

    private double balance;
    // ... constructor and other methods
}

// File: com/mybank/core/AuditService.java
package com.mybank.core;

public class AuditService {
    public void auditAccount(BankAccount account) {
        // CORRECT: AuditService is in the same package as BankAccount,
        // so it can access the 'accountNumber' field.
        System.out.println("Auditing account: " + account.accountNumber);
    }
}

// File: com/mybank/ui/Main.java
package com.mybank.ui;

import com.mybank.core.BankAccount;

public class Main {
    public static void main(String[] args) {
        BankAccount myAccount = new BankAccount(...);
        
        // COMPILE ERROR! 'accountNumber' is not visible here
        // because Main is in a different package (com.mybank.ui).
        // System.out.println(myAccount.accountNumber);
    }
}

- Use Cases:
Framework/Module Design: When you are building a library or a module (e.g., com.mybank.core), you might have several classes that need to interact intimately. You can give them default access to each other's members, while still protecting them from code outside the module.


3. **protected**
- The Family Inheritance
- Core Idea: A protected member is accessible within its own package (like default) AND by subclasses even if they are in a different package.
- How It Works: This is the inheritance-friendly modifier. It signals, "This is an internal detail, but I trust my children (subclasses) to access it." It allows a subclass to directly access or manipulate the internals of its parent class, which is sometimes necessary for extending functionality.

// File: com/mybank/core/BankAccount.java
package com.mybank.core;

public class BankAccount {
    private double balance;
    protected String accountHolderName; // Protected access

    public BankAccount(String name, double initialDeposit) {
        this.accountHolderName = name;
        this.balance = initialDeposit;
    }
    // ... other methods
}

// File: com/mybank/products/SavingsAccount.java (a different package)
package com.mybank.products;

import com.mybank.core.BankAccount;

// SavingsAccount IS-A BankAccount (inherits from it)
public class SavingsAccount extends BankAccount {
    private double interestRate;

    public SavingsAccount(String name, double initialDeposit, double rate) {
        super(name, initialDeposit);
        this.interestRate = rate;
    }

    public void applyInterest() {
        // CORRECT: SavingsAccount is a subclass, so it can access the 
        // protected 'accountHolderName' field from its parent, even
        // though it's in a different package.
        System.out.println("Applying interest for " + this.accountHolderName);
        // ... logic to add interest to the (private) balance via a public method
    }
}

- Use Cases:
Extending Classes: When you design a class to be inherited (a "base class"), you might make certain fields or methods protected to give subclasses the flexibility they need to implement their own behavior.

4. **public**
 - The Open API
- Core Idea: The least restrictive level. A public member is accessible from anywhere. Any other class, in any package, can see and access it.
- How It Works: This defines the public interface of your class. These are the methods and, in rare cases, fields that you want the entire world to be able to use. This is your class's contract with the outside world.

// BankAccount.java
public class BankAccount {
    // ... private fields ...

    // The constructor is PUBLIC so other classes can create BankAccount objects.
    public BankAccount(String name, double initialDeposit) { ... }

    // This method is PUBLIC. It's the primary way for the outside world
    // to interact with the object's state.
    public void deposit(double amount) { ... }

    // This method is PUBLIC. It's a safe way to read the object's state.
    public double getBalance() { ... }
}

- Use Cases:
- Public API Methods: Almost all methods that represent an action an object can perform should be public (e.g., car.startEngine(), user.sendEmail(), list.add(item)).
- Constructors: To be able to create an instance of a class from outside its package, its constructor must be public.
- Constants: It's common to have public static final constants that are globally accessible (e.g., Math.PI).
