# destructor
- A destructor is a special member function of a class that is automatically called when an object is destroyed.
Its purpose is to release resources held by the object (memory, files, locks, sockets, etc.).

class File {
public:
    ~File() {
        // cleanup code
    }
};

**Key points**

Name = ~ClassName

No return type

No parameters

Called automatically

Used for manual resource cleanup

Only one destructor per class

## Does Java have a destructor?

❌ No, Java does NOT have destructors like C++

Why?

Java uses Garbage Collection

Programmer does not control object destruction time

## Destructor Behavior: Stack vs Heap (C++)
1️⃣ Stack Allocation
void func() {
    MyClass obj;
} // destructor called here


How it works

Object is created on the stack

Lifetime is tied to scope

When scope ends → object is destroyed

Destructor is called automatically

📌 Order: Last created → first destroyed (LIFO)

✔️ No delete needed
✔️ Deterministic and safe

# 2️⃣ Heap Allocation
MyClass* obj = new MyClass();
// ...
delete obj;   // destructor called here

How it works

Object is created on the heap

Lifetime is not scope-bound

Destructor is called only when delete is used

📌 Forgetting delete → destructor never runs → memory leak

## Perfect Interview Answer (10 seconds)

For stack-allocated objects, destructors are called automatically when the scope ends. For heap-allocated objects, destructors are called only when delete is explicitly invoked. Java does not support destructors and relies on garbage collection instead.

### why to use delete in case of heap memory

- Heap memory is not tied to scope, so the compiler cannot automatically determine when to free it. Therefore, in C++, we must explicitly call delete to destroy the object and release the memory.

## How to delete a dynamically allocated array (C++)
Correct Syntax
int* arr = new int[5];
delete[] arr;

✔️ Always use delete[] for arrays

# Why delete[] and not delete?

new[] allocates multiple objects

delete[]: Calls the destructor for each element and Frees the entire memory block correctly


# Why doesn’t C++ have a Garbage Collector (GC)?
Short Interview Answer

- C++ does not have a garbage collector because it is designed for deterministic resource management, performance, and full control over memory, which GC cannot guarantee.

1️⃣ Deterministic Destruction (Most Important)

In C++:

{
    File f("data.txt");
} // file closed HERE

Destructor runs exactly at scope exit

Required for:

Files

Locks

Sockets

Hardware resources

📌 GC is non-deterministic → destruction time is unpredictable.

# 4️⃣ RAII Replaces GC in C++

C++ uses RAII (Resource Acquisition Is Initialization):

std::lock_guard<std::mutex> lock(m);


Resource acquired in constructor

Released in destructor

Scope-based cleanup

No GC needed

📌 RAII works for all resources, not just memory.

  **difference_btw_Constructor and Method**

 - Key Interview Differences (One-liners)

. Constructor initializes state; method defines behavior

. Constructor has no return type; method does

. Constructor is auto-invoked; method is explicitly called

. Constructor runs once; methods can run many times

Example in Java
Constructor
class Car {
    int speed;

    // Constructor
    Car(int speed) {
        this.speed = speed;
    }
}

Method
class Car {
    int speed;

    void accelerate() {
        speed += 10;
    }
}
