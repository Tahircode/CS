 **Part 1: Automatic Storage Allocation (The Stack)**
  - e.g className myName;
 - Automatic storage allocation is a mechanism managed entirely by the compiler. It's called "automatic" because you, the programmer, don't have to do anything to request or release the memory. It all happens automatically based on the scope of your code.

- Imagine your program's execution:

- The main() function starts. A "stack frame" containing its local variables is pushed onto the stack.
- main() calls a function, say calculateRoute(). A new stack frame for calculateRoute() is pushed on top of main()'s frame.
- calculateRoute() calls another function, findShortestPath(). Another stack frame is pushed on top.

- When a function finishes executing, its stack frame is popped off the stack, cleanly and instantly destroying all its local variables.

*Features*

- extreamly fast: Allocating memory is just a matter of moving a single pointer (the stack pointer). Deallocating is just moving it back.

- Automatic. The compiler handles everything. No risk of forgetting to free memory.

- Tied to scope:  An object exists only within the {} braces it was declared in. This is predictable and safe

- Size: Limited. The stack has a fixed, relatively small size (e.g., 1-8 MB). Trying to allocate too much (e.g., a massive local array) or having infinite recursion will cause a "Stack Overflow" crash.

- If the stack is a small, organized workbench, the heap is a massive, sprawling warehouse.

- Dynamic storage allocation is for data whose lifetime cannot be determined at compile time. You, the programmer, explicitly request a block of memory from the operating system at runtime. This memory lives in a large pool called the heap (or free store).

- You don't get the memory directly; instead, you get a pointer or reference that tells you where in the warehouse your object is located.

- The process:

Allocation: You use a keyword like new (C++, Java, C#) or a function like malloc() (C). This triggers the memory manager to find a suitable free block in the heap.
Usage: You use the pointer/reference to interact with your object.
Deallocation: You are responsible for releasing the memory when you're done.
- In C++, you must explicitly use delete.
- In Java, C#, and Python, a Garbage Collector (GC) automatically finds and frees memory that is no longer being used.

*Features*
- speed: slower: Allocation requires a complex algorithm to find a free block of the right size (which can lead to fragmentation). Deallocation (especially with a GC) can also be time-consuming.

- Management : Manual or Automatic (GC): This gives you flexibility but also responsibility. Forgetting to delete in C++ causes a memory leak.

- Lifetime: Not tied to scope: An object created on the heap lives until it is explicitly destroyed or collected. It can outlive the function that created it.

- Size: Very large: Limited only by the available RAM and virtual memory of the system. You can create very large objects and millions of them (until you run out of memory).

**Part 4: The Deep Connection to OOP & The Garbage Collector**

- Object Ownership and Lifetime: The choice of stack vs. heap is fundamentally about object lifetime. When you design a system, you must decide: Who creates this object? Who is responsible for destroying it? How long does it need to live?
If an object is only a temporary helper within a method, the stack is perfect.
If you're creating a Player object in a game that must persist for the entire game session, it must be on the heap.

- The Garbage Collector (GC) as an Abstraction: The GC in modern languages is a masterpiece of abstraction. It hides the incredibly complex and error-prone task of manual memory management. As an OOP programmer in Java or C#, you get the benefit of dynamic allocation (flexibility) without the biggest drawback (manual deallocation). You can simply focus on the object's behavior and let the language runtime handle the memory. This is a prime example of how a language feature encapsulates a difficult problem to make you more productive.

- Pointers, References, and Object Graphs: When you create objects on the heap, they communicate with each other via references. This creates a web or "graph" of objects all over the heap. The Garbage Collector's job is to traverse this graph, starting from known "roots" (like active variables on the stack), and find all the reachable objects. Anything it can't reach is considered "garbage" and is cleaned up
