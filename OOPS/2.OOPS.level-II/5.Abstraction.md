# 1. Abstract Class:
- An abstract class is a class that cannot be instantiated and may contain:

. Abstract methods (no body)
. Concrete methods (with implementation)
. Data members
. Constructors

- In C++, abstraction is achieved through abstract classes, which are created using pure virtual functions. C++ does not have a separate interface keyword; an interface is a conceptual design pattern implemented as a special kind of abstract class.

- A class becomes abstract if it has at least one pure virtual function.
 
:: Syntax: virtual void functionName() = 0;

* The = 0 tells the compiler, "This function has no implementation in this base class."
Any class that inherits from this abstract class must provide an implementation for all pure virtual functions, or it will also be abstract

* When to use abstract classes

. When classes share common state + behavior
. When partial implementation is needed

**Early Binding & Late Binding**

- This is a core OOP + low-level concept, and interviewers often link it directly to polymorphism, vtable/vptr, static vs dynamic typing.

# 1️⃣ What is Binding? (Base idea)

Binding means:
- Deciding which function implementation will be executed for a function call.

* 2️⃣ Early Binding (Static Binding)
- Early binding means the function call is resolved at compile time.

The compiler already knows exactly which function to call before the program runs.

When Early Binding Happens:
. Non-virtual functions
. Function overloading
. Static methods

- Compile-time polymorphism

Example (C++)
class A {
public:
    void show() {
        cout << "A::show\n";
    }
};

A a;
a.show();   // early binding


🔹 Compiler directly binds call to A::show()
🔹 No runtime decision
🔹 No vtable involved

Example (Function Overloading)
void add(int a, int b);
void add(double a, double b);

add(2, 3);     // resolved at compile time
add(2.5, 3.5); // resolved at compile time

# 3️⃣ Late Binding (Dynamic Binding)
- Late binding means the function call is resolved at runtime.
- The actual function is chosen based on the object type, not the pointer/reference type.

When Late Binding Happens:
. Virtual functions
. Runtime polymorphism
. Calls via base class pointer/reference

Example (C++)
class A {
public:
    virtual void show() {
        cout << "A::show\n";
    }
};

class B : public A {
public:
    void show() override {
        cout << "B::show\n";
    }
};

A* obj = new B();
obj->show();   // late binding

🔹 Decision happens at runtime
🔹 Uses vtable + vptr
🔹 Calls B::show()

How Late Binding Works Internally:
. Object has a vptr
. vptr points to class vtable
. vtable contains function addresses

**Limitation of Virtual Function**
- slow
  because:
    - normal function call => obj.func(); -=> direct jump
    - virtual function call => obj.func(); -=> vtable -> function pointer -> jump
- difficult to debug
   because:

   Base* b = new Derived();
   b->foo();

   and When debugging:
   - You see Base*
   - But execution jumps to Derived::foo()

   This causes confusion:

   - Why is this function called?
   - Where did this override come from?

# 2. Interfaces
- An interface defines only what a class must do, not how.
- all functions are abstract

class Shape {
       virtual void pay() = 0; 
}

- When to use interfaces
. To define capabilities
. To support multiple inheritance
. To avoid tight coupling

Interfaces in C++ (The Conceptual Approach)
- In C++, an interface is an abstract class that meets the following criteria:

It has no data members (or only static const members).
It has only pure virtual functions.
It has a virtual destructor.
It represents a pure "can-do" relationship.

Example: The ILogger Interface
// This is an INTERFACE in C++. It's just a special kind of abstract class.
class ILogger {
public:
    virtual ~ILogger() = default; // Virtual destructor is a must

    // Pure virtual functions defining the logging contract
    virtual void logInfo(const std::string& message) = 0;
    virtual void logError(const std::string& message) = 0;
};

// A concrete implementation of the ILogger interface
class ConsoleLogger : public ILogger {
public:
    void logInfo(const std::string& message) override {
        std::cout << "[INFO]: " << message << std::endl;
    }
    void logError(const std::string& message) override {
        std::cerr << "[ERROR]: " << message << std::endl;
    }
};



 # Abstraction in JAVA

- Abstraction in Java
Java has explicit, dedicated keywords for both abstract classes and interfaces, making the distinction clearer.

Abstract Classes in Java
You use the abstract keyword to define an abstract class and its abstract methods.

Example: The Shape Abstract Class in Java
public abstract class Shape {
    protected String name;

    public Shape(String name) {
        this.name = name;
    }

    // A regular method with implementation
    public void printName() {
        System.out.println("This is a " + this.name);
    }

    // An abstract method (no body, ends with a semicolon)
    // Any concrete subclass MUST implement this.
    public abstract double getArea();
}

// A concrete subclass
public class Circle extends Shape {
    private double radius;

    public Circle(double radius) {
        super("Circle"); // Call the parent constructor
        this.radius = radius;
    }

    @Override // Annotation to verify we are correctly overriding
    public double getArea() {
        return Math.PI * radius * radius;
    }
}

# Interfaces in JAVA

* Interfaces in Java
- Java has a dedicated interface keyword. An interface is a pure contract.

- Key Characteristics of Java Interfaces (pre-Java 8):

. All methods are public and abstract by default.
. All fields are public, static, and final (constants) by default.
. A class can implement multiple interfaces.
Example: The ILogger Interface in Java

// The 'interface' keyword makes the intent explicit.
public interface ILogger {
    // Methods are public and abstract by default
    void logInfo(String message);
    void logError(String message);
}

// A class implements the interface using the 'implements' keyword
public class ConsoleLogger implements ILogger {
    @Override
    public void logInfo(String message) {
        System.out.println("[INFO]: " + message);
    }

    @Override
    public void logError(String message) {
        System.err.println("[ERROR]: " + message);
    }
}

## When to Use Which? (The Universal Rule)
This is the most important practical takeaway.

* Use an Abstract Class when:

- You want to share code among several closely related classes.
- The classes that inherit from it have a strong "is-a" relationship (e.g., Dog, Cat, Lion are all Animals).
- You need to define non-static or non-final fields. This allows you to specify a common state for all subclasses.

* Use an Interface when:

- You expect that unrelated classes will implement your interface. A Car and a Bird are very different, but they could both implement a Flyable interface.
- You want to specify the behavior of a particular data type, but you are not concerned about who implements its behavior.
- You want to take advantage of multiple inheritance of type.
**In short: Abstract classes are for things. Interfaces are for behaviors.**
