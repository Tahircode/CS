# Encapsulation
- Encapsulation is wrapping data and methods together and restricting direct access to data.
- why? : to hide information, for security, to make class as read only

- class itself is an example of encapsulation
 putting data inside a class and giving access away to only specific data.

what is a fully encapsulated class:

- A fully encapsulated class is a class in which all instance variables are declared private and are accessed or modified only through public getter and setter methods.

 👉 Achieved using access modifiers.
```
class User {
    private int age;

    public int getAge() {
        return age;
    }

    public void setAge(int age) {
        if (age >= 0)
            this.age = age;
    }
}
```

Key idea: Data is hidden, Access is controlled

# 2️⃣ Encapsulation — Intermediate Level
Why encapsulation is needed
- Data protection
- Validation
- Maintainability
- Controlled state changes

Example (bad vs good):

❌ Without encapsulation

user.age = -10;   // invalid state


✅ With encapsulation

user.setAge(-10); // rejected safely

📌 Encapsulation ensures the object always remains valid.

# 3️⃣ Encapsulation — Design Level

At this level, encapsulation is NOT just getters/setters.

Encapsulation = Controlling behavior, not exposing data

❌ Bad design (Anemic model):

user.setBalance(user.getBalance() - 100);

✅ Good design:

user.withdraw(100);

Why?

Business rules stay inside the object

External code cannot break invariants

📌 Advanced definition

Encapsulation means exposing behavior, not internal state.

# 4️⃣ Encapsulation — Advanced / System Level

Encapsulation of change

Encapsulation protects your system from future changes.
```
class Cache {
    private Map<String, String> store;
}
```

Later you change implementation:

Map → Redis → Database


✔️ External code remains unchanged
✔️ Only internal implementation changes

📌 This is real-world encapsulation

# 5️⃣ Encapsulation & Thread Safety

Encapsulation helps enforce synchronization rules.
```
class Counter {
    private int count;

    public synchronized void increment() {
        count++;
    }
}
```
✔️ Thread safety is guaranteed
❌ Without encapsulation → race conditions  

## Encapsulation Example – C++
```
#include <iostream>
using namespace std;

class BankAccount {
private:
    double balance;   // data hidden

public:
    BankAccount(double initialBalance) {
        if (initialBalance < 0)
            balance = 0;
        else
            balance = initialBalance;
    }

    // behavior exposed, not data
    void deposit(double amount) {
        if (amount > 0)
            balance += amount;
    }

    void withdraw(double amount) {
        if (amount <= 0) return;
        if (amount > balance) {
            cout << "Insufficient balance\n";
            return;
        }
        balance -= amount;
    }

    double getBalance() const {
        return balance;   // read-only access
    }
};
```
- Why this is proper encapsulation (C++)

. balance is private
. No direct modification from outside
. Business rules are enforced inside the class
. Object can never be in an invalid state
.

## Encapsulation Example – Java
```
class BankAccount {
    private double balance;   // data hidden

    public BankAccount(double initialBalance) {
        if (initialBalance < 0)
            balance = 0;
        else
            balance = initialBalance;
    }

    // behavior exposed, not data
    public void deposit(double amount) {
        if (amount > 0)
            balance += amount;
    }

    public void withdraw(double amount) {
        if (amount <= 0) return;
        if (amount > balance) {
            System.out.println("Insufficient balance");
            return;
        }
        balance -= amount;
    }

    public double getBalance() {
        return balance;   // read-only access
    }
}
```
- Why this is proper encapsulation (Java)
. State is fully protected
. Only meaningful operations are exposed
. Internal logic can change safely
. External code cannot break invariants
