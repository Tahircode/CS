# Inheritance

## 1️⃣ What is Inheritance? (Definition)

Inheritance is an OOP mechanism where a child class (derived class) acquires properties and behavior of a parent class (base class) and may extend or modify them.

📌 Represents an IS-A relationship

 Inheritance allows a class to acquire and extend the behavior of another class, enabling code reuse, extensibility, and polymorphism.

## 2️⃣ Why do we need Inheritance?

Main reasons
- Code Reuse
- Extensibility
- Polymorphism
- Logical hierarchy

Without inheritance:
- Code duplication
- Difficult maintenance
- Poor scalability

## 4️⃣ How Inheritance Works (Syntax)
C++
```
class Car : public Vehicle {};
```
Java
```
class Car extends Vehicle {}
```
## 5️⃣ What Inheritance Gives You
Child class gets
- Parent data members
- Parent member functions
- Ability to override behavior
```
class Car extends Vehicle {
    void move() {
        System.out.println("Car moves on road");
    }
}
```

## 6️⃣ When to Use Inheritance (IMPORTANT)

Use inheritance only when:
- ✔️ There is a true IS-A relationship
- ✔️ Subclass is a specialized version of parent
- ✔️ Behavior can be reused meaningfully

## 7️⃣ When NOT to Use Inheritance (Very Important)

❌ Do NOT use inheritance when:
- Relationship is HAS-A
- Only code reuse is needed
- Base class changes often

📌 Prefer composition over inheritance

### ✅ Inheritance – C++ (Good Design)
```
class Vehicle {
protected:
    int speed;

public:
    Vehicle(int speed) : speed(speed) {}

    virtual void move() const {
        cout << "Vehicle is moving\n";
    }

    virtual ~Vehicle() {}   // important
};

class Car : public Vehicle {
public:
    Car(int speed) : Vehicle(speed) {}

    void move() const override {
        cout << "Car moves at " << speed << " km/h on road\n";
    }
};
```
Usage:

Vehicle* v = new Car(80);

v->move();

delete v;

### ✅ Inheritance – Java (Good Design)
```
abstract class Vehicle {
    protected int speed;

    Vehicle(int speed) {
        this.speed = speed;
    }

    abstract void move();
}

class Car extends Vehicle {
    Car(int speed) {
        super(speed);
    }

    @Override
    void move() {
        System.out.println("Car moves at " + speed + " km/h on road");
    }
}
```
Usage
Vehicle v = new Car(80);
v.move();

### Modes of Inheritance

| Base class member | Public inheritance | Protected inheritance | Private inheritance |
| ----------------- | ------------------ | --------------------- | ------------------- |
| **public**        | public             | protected             | private             |
| **protected**     | protected          | protected             | private             |
| **private**       | ❌ not accessible   | ❌ not accessible     | ❌ not accessible   |


**1️⃣ Public Inheritance (most common)**
```
class Base {
public:
    int a;
protected:
    int b;
private:
    int c;
};

class Derived : public Base {
    void test() {
        a = 10; // ✔ public → public
        b = 20; // ✔ protected → protected
        // c ❌ not accessible
    }
};
```
👉 IS-A relationship
Used when Derived is a Base.

**2️⃣ Protected Inheritance**
```
class Derived : protected Base {
    void test() {
        a = 10; // ✔ public → protected
        b = 20; // ✔ protected → protected
        // c ❌
    }
};
```
👉 Outside code cannot access a anymore
Used internally or for framework design.


**3️⃣ Private Inheritance**
```
class Derived : private Base {
    void test() {
        a = 10; // ✔ public → private
        b = 20; // ✔ protected → private
        // c ❌
    }
};
```
👉 NOT an IS-A relationship
Used for implementation reuse only

**🧠 One-line Memory Trick (Interview GOLD)**

Public inheritance keeps access same,
Protected inheritance downgrades public → protected,
Private inheritance downgrades everything → private.
Private base members are never accessible.


**🎯 C++ vs Java (important insight)**

| Inheritance modes   | public / protected / private | ❌ only `extends` |
| Default mode        | private (class)              | public           |
| Private inheritance | ✔ allowed                    | ❌ not allowed    |

 ## Diamond problem
```
  
               A (Base)
              /  \
   (parent2) B    C (parent1)
              \  /
                D
             (child)
```
- The Diamond Problem occurs when a class inherits from two classes (let's call them Parent1 and Parent2), and both of those classes inherit from a single common Grandparent class. This creates a diamond-shaped inheritance hierarchy.
```
class Device {
private:
    int deviceId;
public:
    Device() : deviceId(0) { std::cout << "Device constructor\n"; }
    void powerOn() { std::cout << "Device " << deviceId << " is ON\n"; }
};

class Scanner : public Device {
public:
    Scanner() { std::cout << "Scanner constructor\n"; }
    void scan() { std::cout << "Scanning\n"; }
};

class Printer : public Device {
public:
    Printer() { std::cout << "Printer constructor\n"; }
    void print() { std::cout << "Printing\n"; }
};

class Copier : public Scanner, public Printer {
public:
    Copier() { std::cout << "Copier constructor\n"; }
};
```
- This creates two distinct, deep-seated problems:

**Problem 1: The Ambiguity (The Obvious Problem)**

- If you create a Copier object and try to call a method from the Device class, the compiler doesn't know which path to take.
```
int main() {
    Copier myCopier;
    myCopier.powerOn(); // AMBIGUOUS CALL!
    return 0;
}
```

Compiler Error:

error: request for member 'powerOn' is ambiguous
myCopier.powerOn();
note: candidates are: void Device::powerOn()
note: candidates are: void Device::powerOn()

- The compiler sees that myCopier has a Scanner subobject which has a Device subobject with a powerOn() method. It also sees a Printer subobject which has its own Device subobject with a powerOn() method. Which one should it call? It can't decide.

**Problem 2: The Duplicated Grandparent (The Deeper Problem)**

- Memory Layout of a Copier object (without virtual inheritance):
```
+------------------------------------------+
|            Copier Object                  |
+------------------------------------------+
| Scanner Subobject                         |
|   +------------------------------------+  |
|   | Device Subobject (for Scanner)     |  |
|   |   - deviceId                       |  |
|   +------------------------------------+  |
|   - Scanner-specific data              |  |
+------------------------------------------+
| Printer Subobject                         |
|   +------------------------------------+  |
|   | Device Subobject (for Printer)     |  |
|   |   - deviceId                       |  |
|   +------------------------------------+  |
|   - Printer-specific data              |  |
+------------------------------------------+
| - Copier-specific data                    |
+------------------------------------------+
```
- This is the more insidious issue. The Copier object doesn't just contain one Device; it contains two separate Device subobjects.

- Consequences of this duplication:

. Bloated Objects: The object is larger than necessary because it contains two copies of the Device's data.
. Logical Inconsistency: A Copier is a single physical device. It shouldn't have two separate deviceIds. It has one identity, one power state. The object model no longer accurately reflects reality.

**Part 3: The Solution - Virtual Inheritance**

- Virtual inheritance is a C++ mechanism designed specifically to solve the diamond problem. You use it by adding the virtual keyword to the inheritance declaration of the middle classes.

- The Promise: When you declare class Scanner : virtual public Device, you are making a promise to the compiler: "If any other class also virtually inherits from Device, and a class eventually inherits from both of us, please ensure there is only one shared Device subobject at the top."

**How It Works (The Deep Mechanism):**

- The compiler achieves this by changing the object's memory layout. Instead of embedding a Device subobject inside Scanner and Printer, it makes Scanner and Printer hold a pointer or offset to the single, shared Device subobject, which will be constructed at the top level of the most-derived class (Copier).

- This is often implemented using a Virtual Base Table (vbtable).

. The Scanner and Printer objects now contain a hidden pointer (a vbptr - virtual base pointer) that points to their respective vbtable.
. The vbtable contains an offset that tells the object how to find the location of the shared Device subobject from the current location.

*Memory Layout of a Copier object (with virtual inheritance):*
```
+------------------------------------------+
|            Copier Object                  |
+------------------------------------------+
| Shared Device Subobject (at the top)      |
|   - deviceId                             |  <-- ONLY ONE COPY!
+------------------------------------------+
| Scanner Subobject                         |
|   - vbptr --> points to Scanner's vbtable |
|   - (vbtable contains offset to Device)   |
|   - Scanner-specific data                |  |
+------------------------------------------+
| Printer Subobject                        |
|   - vbptr --> points to Printer's vbtable |
|   - (vbtable contains offset to Device)   |
|   - Printer-specific data               |  |
+------------------------------------------+
| - Copier-specific data                   |
+------------------------------------------+
```
- Now, when myCopier.powerOn() is called, both the Scanner and Printer paths can use their vbptr and vbtable to correctly locate the single, shared Device subobject. The ambiguity is resolved.

- The virtual Copier will be smaller than the non-virtual one because it doesn't contain two copies of the Device's data members. It will be slightly larger than a Scanner or Printer alone due to the added vbptr.

- Ambiguity in the diamond problem is solved using virtual inheritance, not by v-tables or the scope resolution operator.
```
class A {
public:
    int x;
};

class B : virtual public A {};
class C : virtual public A {};
class D : public B, public C {};
```
Now:
```
D obj;
obj.x;   // ✅ NO ambiguity
```
What virtual inheritance does:

- Ensures only ONE shared instance of A
- B and C both refer to the same A
- Eliminates duplication at the memory level

Under the Hood (important): 

 Virtual inheritance adds:
- hidden pointers (often called vbptr)
- indirection to locate the shared base

This is NOT the same as v-table

- v-table → virtual functions
- vbptr → virtual base classes

Constructor Rule (very important ⚠️)

When using virtual inheritance:
```
class D : public B, public C {

public:

    D() : A(10) {}  // MOST derived class initializes A

};
```
👉 The most derived class initializes the virtual base.

4. ## Hierarchical Inheritance
```
        Parent
```

Hierarchical inheritance is when multiple derived classes inherit from a single base class, sharing common behavior while providing specialized implementations.
```
class Shape {

class Circle : public Shape {
public:
    void area() {
        cout << "Area of circle\n";
    }

class Rectangle : public Shape {
public:

    }
};

int main() {
    Circle c;
    Rectangle r;

    c.draw(); // inherited
    r.draw(); // inherited

    c.area();
    r.area();
}
```

## IN JAVA

q: Can we create multiple inheritance in Java?

- No, not of classes (implementation/state).

q: why not?

- java's creators deliberately chose to avoid multiple inheritance of classes to prevent the "Diamond Problem," which we discussed in the context of C++. This problem creates ambiguity and complexity when a class inherits from two classes that both inherit from a common grandparent

// Hypothetical code - this will NOT compile in Java
```
class Device {
    void powerOn() { System.out.println("Device ON"); }
}

class Scanner extends Device {
    void scan() { System.out.println("Scanning"); }
}

class Printer extends Device {
    void print() { System.out.println("Printing"); }
}
```

// What if this were allowed?

class Copier extends Scanner, Printer { // ERROR in Java
    // ...
}

If you could create a Copier object and call myCopier.powerOn(), which powerOn() method should be called? The one from the Scanner path or the one from the Printer path? The compiler has no way to decide, leading to ambiguity. To avoid this entire category of problems, Java simply forbids it.

**so what is the solutoin**

 Multiple Inheritance of Type with Interfaces
- A class can extend only one class (its parent), but it can implement multiple interfaces.

Example: A Bird
A bird is an Animal, but it can also Fly and it can Sing. These are capabilities, not types of animals.
// The single parent class for state and shared behavior
```
abstract class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public abstract void eat();
}

// A capability (an interface)
interface Flyable {
    void fly();
}

// Another capability (an interface)
interface Singable {
    void sing();
}
```
// The concrete class
```
public class Robin extends Animal implements Flyable, Singable {

    public Robin(String name) {
        super(name); // Call the one parent constructor
    }

    // Implement the abstract method from the parent class
    @Override
    public void eat() {
        System.out.println(name + " is eating worms.");
    }

    // Implement the contract from the Flyable interface
    @Override
    public void fly() {
        System.out.println(name + " is flying gracefully.");
    }

    // Implement the contract from the Singable interface
    @Override
    public void sing() {
        System.out.println(name + " is singing a beautiful song.");
    }
}
```
* The Modern Complication - default Methods (Java 8+)

The introduction of default methods in Java 8 interfaces slightly complicated this picture. default methods allow you to provide a default implementation for a method directly within the interface.

This means interfaces can now have both a contract and some shared implementation.

Re-introducing the Diamond Problem?

What if two interfaces you implement both have the same default method?
```
interface Flyable {
    default void move() {
        System.out.println("Flying through the air.");
    }
}

interface Walkable {
    default void move() {
        System.out.println("Walking on the ground.");
    }
}

public class Duck extends Animal implements Flyable, Walkable {
    // ...
    // myDuck.move(); // AMBIGUOUS! Which one should be called?
}
```
* Java solves this by forcing the implementing class to resolve the ambiguity itself. You must @Override the conflicting method and explicitly choose which implementation you want, or provide a completely new one.

* Resolution:

- public class Duck extends Animal implements Flyable, Walkable {
    public Duck(String name) { super(name); }

    @Override
    public void eat() { /* ... */ }

    // We MUST override to resolve the conflict
    @Override
    public void move() {
        // We can choose one implementation
        System.out.print("As a duck, I am ");
        Flyable.super.move(); // Explicitly call Flyable's version
    }
}

- By using InterfaceName.super.methodName(), you can unambiguously select which parent interface's default method you want to use.
