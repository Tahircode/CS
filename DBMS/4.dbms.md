#### ACID properties

ACID is a set of four properties that guarantee reliable and consistent execution of database transactions.
- A → Atomicity  
- C → Consistency  
- I → Isolation  
- D → Durability

#### Transaction:

A transaction is a group of operations treated as one unit.

OR

A Transaction is a sequence of one or more database operations (like INSERT, UPDATE, DELETE) that:
- Are executed as a single logical unit of work
- Must either complete fully or have no effect at all

* Operations:

1. COMMIT
- Transaction completed successfully & Changes are permanently saved
- COMMIT;

2. ROLLBACK
- Transaction failed & All changes are undone
- ROLLBACK;

3. SAVEPOINT (Optional)
- Used to: Rollback partially
example:
- SAVEPOINT sp1;
- ROLLBACK TO sp1;

* 🧠 Transaction States (VERY IMPORTANT)

- 1️⃣ Active → Transaction is executing
- 2️⃣ Partially Committed → Last statement executed
- 3️⃣ Committed → Successfully completed
- 4️⃣ Failed → Error occurred
- 5️⃣ Aborted → Rolled back

## Atomicity:

its 0 or 1 => A transaction is all or nothing.

All operations execute

OR

None execute (rollback)

 **how to implement atomicity**

DBMS uses three main mechanisms:

- 1️⃣ Logging (Write-Ahead Logging)
- 2️⃣ Undo Mechanism
- 3️⃣ Recovery System

1. Write-Ahead Logging (WAL) ⭐⭐⭐

Before modifying actual database data, DBMS first writes the change to a LOG file. This is called Write-Ahead Logging.

🧠 Example

If updating balance:

Old value = 5000, 
New value = 4000

Log entry stored:

Transaction T1
Old value: 5000
New value: 4000

Only after logging → actual data updated.

why? because if crash happens, DBMS checks log and uses old value to undo

2. Undo Mechanism ⭐

If transaction fails:

DBMS:
- Reads log
- Restores old values
- Rolls back all operations

This ensures:
- All or Nothing

3. Recovery Manager ⭐⭐⭐

DBMS has a Recovery Manager component.

Its job:
- Detect crash
- Use logs
- Undo incomplete transactions
- Redo committed transactions if needed

#### undo vs redo
- UNDO is the process of reversing the effects of an uncommitted (incomplete) transaction.
👉 It restores the database to the old value.

- REDO is the process of reapplying the effects of a committed transaction after a crash.
👉 It ensures the new value is preserved.

| Situation                           | Action |
| ----------------------------------- | ------ |
| Transaction failed                  | UNDO   |
| Transaction committed but not saved | REDO   |

| Feature      | UNDO                          | REDO                          |
| ------------ | ----------------------------- | ----------------------------- |
| Purpose      | Cancel incomplete transaction | Reapply committed transaction |
| Uses         | Old value                     | New value                     |
| Applied When | Transaction NOT committed     | Transaction committed         |
| Ensures      | Atomicity                     | Durability                    |

## 2. Consistancy

Consistency ensures:
- The database moves from one valid state to another valid state, DB remains consistant

example:
before transaction:

A: 200
B: 100
total: 300

after transaction

A: 150
B: 150
total: 300

Total remains valid. Constraints not violated.

- DB must be consistant before & after transaction.

## 3. ISOLATION
- isolation ensures multiple transaction occur independently without interference

🧠 Example

Two users booking last seat:

Without isolation:
- Both may book same seat ❌

With isolation:
- One succeeds
- One fails

🎯 Controlled by:
- Locks
- Isolation levels
- Concurrency control

## 4. Durability

it ensures: 
- Once a transaction is committed, it remains permanent even if system crashes.

🧠 Example

After successful transfer:
- Even if power failure happens,
- Data is stored permanently.

🎯 Achieved by:
- Logging
- Write-ahead logs
- Backup & recovery

in simple words: 
- After commit, even if failure happens, the committed changes are recovered using logs.

## Concurrency
- Multiple transactions executing at the same time (simultaneously) in a database system.
- It allows many users to access and modify data simultaneously.

Transaction: T1, T2, T3,...TN

1. Advantages:

| Advantage            | Explanation                      |
| -------------------- | -------------------------------- |
| Better Performance   | CPU and disk used efficiently    |
| Faster Response Time | Users don’t wait for others      |
| Higher Throughput    | More transactions per second     |
| Multi-user Support   | Essential for real-world systems |


2. Disadvantages:

a. Lost Update Problem
- Two transactions update same data.

b. Dirty Read
- One transaction reads uncommitted data of another.
- If second transaction rolls back → first transaction has incorrect data.
      
c. Non-Repeatable Read
- A row read twice gives different results.

d. Phantom Read
- New rows appear during transaction execution.

#### How to Resolve Concurrency Problems

DBMS uses Concurrency Control Mechanisms.

1. Lock-Based Protocol (Most Common) ⭐

Types of Locks:
- Shared Lock (Read Lock)
- Exclusive Lock (Write Lock)

Prevents:
- Simultaneous conflicting updates

2. Two-Phase Locking (2PL) ⭐⭐⭐

Transaction must:
- Acquire all locks (Growing phase)
- Release locks (Shrinking phase)

Guarantees serializability.

3. Timestamp Ordering
- Transactions executed based on timestamps.

4. MVCC (Multi-Version Concurrency Control)

Used in:
- PostgreSQL
- MySQL (InnoDB)

Creates multiple versions of data instead of locking everything.

5. Isolation Levels ⭐

Controlled using:
- Read Uncommitted
- Read Committed
- Repeatable Read
- Serializable

Higher isolation → more safety → less concurrency.

##### How to Use Concurrency Properly

1️⃣ Use proper isolation levels
2️⃣ Use transactions correctly
3️⃣ Avoid long-running transactions
4️⃣ Use indexing
5️⃣ Design schema properly

⚠️ Important Concept

Concurrency improves performance BUT:

- More concurrency → More risk
- More isolation → Less performance

There is always a trade-off.

## DeadLocks

![alt text](<0.public/image copy 11.png>)

A Deadlock is a situation where:

- Two or more transactions are waiting for each other to release locks, and none of them can proceed.

👉 In simple words:
- Each transaction is waiting for a resource held by the other.
- So the system gets stuck.

🧠 Critical Example

Consider two transactions:

- T1 locks Resource A
- T2 locks Resource B

Now:

- T1 wants Resource B (held by T2)
- T2 wants Resource A (held by T1)

Both are waiting for each other.

Neither can continue.

That is a deadlock.

#### Necessary Conditions for Deadlock ⭐⭐⭐

Deadlock occurs only if ALL 4 conditions hold:

1️⃣ Mutual Exclusion
- Resource can be used by only one transaction at a time.

2️⃣ Hold and Wait
- Transaction holds a resource and waits for another.

3️⃣ No Preemption
- Resource cannot be forcibly taken away.

4️⃣ Circular Wait
- Transactions form a cycle of waiting.

If even one condition is removed → No deadlock.

##### ❌ Problems Caused by Deadlock
- Transactions stuck forever
- Reduced throughput
- Wasted system resources
- System freeze

#### 🛡 How to Handle Deadlock

There are 3 major approaches:

1️⃣ Deadlock Prevention

Break one of the four conditions.

Example:
- Force resource ordering
- No hold-and-wait allowed

2️⃣ Deadlock Avoidance

System checks before granting locks.

Example:
- Banker’s Algorithm (conceptually)

3️⃣ Deadlock Detection & Recovery ⭐ (Most Practical)
Step 1: Detect

Use Wait-For Graph.

If cycle exists → deadlock.

Step 2: Recover
- Abort one transaction
- Release its locks
- Restart it


#### advanced deadlock prevention techniques

#### Wait-Die scheme & Wound-Wait scheme:

Wait-Die and Wound-Wait are timestamp-based deadlock prevention schemes where transactions are either allowed to wait or forced to abort based on age.

| Feature             | Wait-Die        | Wound-Wait        |
| ------------------- | --------------- | ----------------- |
| Based On            | Timestamp       | Timestamp         |
| Older transaction   | Waits           | Preempts (wounds) |
| Younger transaction | Aborts          | Waits             |
| Aggressiveness      | Less aggressive | More aggressive   |
| Rollbacks           | More frequent   | Fewer rollbacks   |

examples:

wait-die:

T1 (older) wants resource held by T2 (younger)

→ T1 waits ✅

T2 (younger) wants resource held by T1 (older)

→ T2 aborts ❌

Deadlock avoided because younger never waits for older.

wound-wait

T1 (older) wants resource held by T2 (younger)

→ T2 aborts ❌

T2 (younger) wants resource held by T1 (older)

→ T2 waits ✅
