## Functional dependency
A functional dependency is a relationship between attributes where one attribute uniquely determines another.

example:
- a depends on b as: a = b^3.
     - 1 -> 1
     - 2 -> 8
     - 3 -> 27
- if 2 = 8 and also 2 = 9 , it wont satisfy the condition of uniqueness
- if 3 = 27 and also 4 = 27 it wont satisfy the condition of uniqueness

### Types of FD

#### 1. Trivial Functional Dependency
- a dependency where:
     - Y ⊆ X
- example:
      - {RollNo, Name} → Name

#### 2. Non-Trivial Functional Dependency
- Y is not a subset of X.
- example:
    - RollNo → Name

#### 3. Completely Non-Trivial FD
X ∩ Y = ∅

example:
- RollNo → DeptName

#### 4. Full Functional Dependency ⭐

Y depends on the entire composite key, not part of it.

example:
- ENROLLMENT
     - | StudentID | CourseID | Grade |
     - Composite key = (StudentID, CourseID)
- Grade depends on both together.

#### 5. Partial Dependency ⭐ (Important for 2NF)

Y depends on part of a composite key.

Example 
- if:
    - (StudentID, CourseID) → StudentName
- but
    - StudentID → StudentName
- Then it is partial dependency.

#### 6. Transitive Dependency ⭐⭐⭐ (Important for 3NF)

if:
- A → B
- B → C
Then
- A → C

Example:
- RollNo → DeptID
- DeptID → DeptName
Therefore:
- RollNo → DeptName
This is transitive dependency.

## Normalization

Normalization is the process of organizing relations to reduce redundancy and eliminate anomalies using functional dependencies.

anomalies can be: insertion anomaly, deletion anomaly, update anomaly

| Anomaly   | Problem                        |
| --------- | ------------------------------ |
| Insertion | Cannot insert independent data |
| Update    | Multiple updates required      |
| Deletion  | Accidental data loss           |

Anomaly example:

Suppose:
- We want to insert a new department:
    - DeptID = D3
    - DeptName = ME
    - Location = Block C

But no student exists yet.

❌ We cannot insert department without RollNo & Name.

This is insertion anomaly.

Update Example:

if:
- DeptLocation changes from Block A → Block Z
- we must update the same data in multiple rows.
- If we forget one row → inconsistency.

Deletion example:
- Occurs when deleting a record unintentionally removes useful information.
If:
- Ali (last CS student) leaves:

We delete:

| 102 | Ali | D1 | CS | Block A |

- If Ali was the only CS student,
- we lose department information entirely.

❌ Department data lost.

## Different forms of Normalization
Normalization progresses step-by-step:
- 1NF → 2NF → 3NF → BCNF → 4NF → 5NF

#### 1. First Normal Form (1NF)


A table is in 1NF if:
- All values are atomic (indivisible)
- No repeating groups
- No multi-valued attributes

Example:

from:

| RollNo | Name  | PhoneNumbers |
| ------ | ----- | ------------ |
| 101    | Tahir | 9876, 8765   |

to:

| RollNo | Name  | PhoneNumber |
| ------ | ----- | ----------- |
| 101    | Tahir | 9876        |
| 101    | Tahir | 8765        |
- Multi-valued attributes.

#### 2. Second Normal Form (2NF)

A table is in 2NF if:
- It is already in 1NF
- No partial dependency (FD: xy -> z & y -> z)
- Every non-prime attribute fully depends on entire composite key

example:

Composite Key: (StudentID, CourseID)

if:

StudentID → StudentName
- This is partial dependency.

Fix:
- Split into:
      - STUDENT(StudentID, StudentName)
      - ENROLLMENT(StudentID, CourseID)

it removes Partial dependency.

#### 3. Third Normal Form (3NF)

A table is in 3NF if:
- It is in 2NF
- No transitive dependency

If:

RollNo → DeptID
DeptID → DeptName

Then:

RollNo → DeptName (transitive)

Fix:

Split into:
- STUDENT(RollNo, DeptID)
- DEPARTMENT(DeptID, DeptName)

🎯 Removes:

Transitive dependency.

#### 4. Boyce-Codd Normal Form (BCNF) ⭐

3NF + For every FD:
- X → Y

X must be a super key.

Why BCNF?
- 3NF may still allow certain anomalies when:
    - Multiple candidate keys exist.
- BCNF is stricter than 3NF.

🎯 Removes:
- Dependency where determinant is not a super key.

#### 5. Fourth Normal Form (4NF)

A table is in 4NF if:
- It is in BCNF
- No multi-valued dependency (MVD)

Example

If Student has:
- Multiple skills
- Multiple hobbies

But skills and hobbies are independent.

Split into:
- STUDENT_SKILLS
- STUDENT_HOBBIES

🎯 Removes:
- Multi-valued dependency.

#### 6. Fifth Normal Form (5NF)

A table is in 5NF if:
- It is in 4NF
- No join dependency

Used when:

- Data can be reconstructed from smaller tables without loss.

Very rare in practical exams.


| Normal Form | Removes                 |
| ----------- | ----------------------- |
| 1NF         | Multi-valued attributes |
| 2NF         | Partial dependency      |
| 3NF         | Transitive dependency   |
| BCNF        | Non-superkey dependency |
| 4NF         | Multi-valued dependency |
| 5NF         | Join dependency         |
