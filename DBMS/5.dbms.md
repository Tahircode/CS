## partitioning

One big table is split into smaller physical pieces for better performance and management.

conditions:
- table-size > 2GB
- Predictable Query Patterns
     - Queries frequently filter on a specific column.
     - That column can act as a partition key.
     - example: date, region, customerId
- Clear Partition Key
    - A good partition key must:
         - Be frequently used in WHERE clause.
         - Have evenly distributed values.
         - Not cause data skew.
    - Bad partition key → one partition becomes huge.
- Independent Data Segments
      -  Partitioning is effective when:
             - Data can logically be divided.
             - Partitions are mostly accessed independently.

      - Example:
             - Year-wise sales
             - Region-wise customers

- Maintenance Requirements
     -  Partitioning helps when:
          - Old data must be archived.
          - Backup must be done per partition.
          - Data deletion should be quick.
     - Example:
          - Drop partition for 2019 instead of deleting millions of rows.

- Parallel Processing Needed
     - Partitioning allows:
          - Parallel queries.
          - Multiple CPUs to work simultaneously.
     - Used in:
          - Data warehouses
          - Big analytics systems
- Minimal Cross-Partition Joins
      - Partitioning works best when:
          - Queries mostly operate within one partition.
          - Cross-partition joins are rare.
       - Frequent cross-partition joins reduce benefit.

#### Its Benefits
- quick access
- availability
- manageable data

## 🧩 Types of Partitioning

#### 1️⃣ Horizontal Partitioning ⭐

Rows are divided across partitions.

Each partition contains different rows but same columns.

Example

ORDERS table partitioned by year:
- Partition 1 → Orders from 2022
- Partition 2 → Orders from 2023
- Partition 3 → Orders from 2024

Same structure, different rows.

#### 2️⃣ Vertical Partitioning

Columns are divided into different tables.

Each partition contains fewer columns.

Example

CUSTOMER table:

| CustomerID | Name | Address | Phone | Email |

Split into:
- Customer_Basic(CustomerID, Name)
- Customer_Contact(CustomerID, Phone, Email)

#### 3️⃣ Range Partitioning ⭐

Data partitioned based on value ranges.

Example:

0–1000

1001–2000

Commonly used with dates.

#### 4️⃣ List Partitioning

Data partitioned based on predefined list values( e.g region ).

Example:

Region = 'North'

Region = 'South'

#### 5️⃣ Hash Partitioning

Data distributed using hash function.

Example:
- partition = hash(CustomerID) % 4

Used for even distribution.

## Single-Level Partitioning

Partitioning a table using one partitioning method only.

The table is divided into partitions based on a single strategy.

types:
- Range Partitioning
- List Partitioning
- hash Partitioning
- Key Partitioning
    -  Similar to hash but DBMS internally decides hash function.
    - Used in:
        - Oracle
        - MySQL

## 🔷 Composite (Multi-Level) Partitioning ⭐⭐⭐

Composite partitioning uses two partitioning techniques together.

Also called:
- Multi-Level Partitioning
- Subpartitioning

🧠 Critical Example

Suppose we have:

ORDERS(OrderID, CustomerID, OrderDate, Region, Amount)

If we partition:

Only by OrderDate → Single-Level

By OrderDate AND inside each year by Region → Composite-Level

##### 🧩 Types of Composite Partitioning

1️⃣ Range–Hash Partitioning ⭐
 - First by range
 - Then by hash inside each range

Common in:
- Large data warehouses

2️⃣ Range–List Partitioning
- First by date
- Then by region

3️⃣ List–Hash Partitioning
- First by region
- Then by hash inside region

4️⃣ Range–Range Partitioning
- Range inside range (less common)

🎯 When to Use Composite Partitioning

Used when:
- Table is extremely large
- Multiple filtering columns are common
- Need parallel processing
- Need better load balancing
- Avoid data skew

use PARTITION BY

## Sharding

Sharding is a technique of horizontally splitting a database into multiple smaller databases (called shards) and distributing them across different servers.

Each shard stores a subset of the data, but all shards together represent the full dataset.

👉 In simple words:
  - Instead of one huge database server, the data is split across multiple servers.


🧠 Critical Example

Imagine a USER table with 100 million users.

Instead of storing all users in one database:

- Shard 1 (Server A) → Users with ID 1–25M
- Shard 2 (Server B) → Users with ID 25M–50M

Now queries are distributed across servers, reducing load.

#### 🧩 Types of Sharding

1️⃣ Range-Based Sharding

Data is divided based on value ranges.

Shard 1 → UserID 1–1000
Shard 2 → UserID 1001–2000

✔ Simple
❌ Can cause uneven load (hotspots).

2️⃣ Hash-Based Sharding ⭐

Data is distributed using a hash function.

Example:

Shard = hash(UserID) % 4

✔ Even distribution
❌ Harder to rebalance.

3️⃣ Directory-Based Sharding

A lookup table determines which shard stores the data.

Example:

UserID → Shard mapping table

✔ Flexible
❌ Extra overhead.

#### 3️⃣ Difference Between Sharding and Partitioning

| Feature       | Partitioning             | Sharding                         |
| ------------- | ------------------------ | -------------------------------- |
| Location      | Same database server     | Multiple servers                 |
| Purpose       | Manage large tables      | Scale across machines            |
| Data split    | Logical split            | Physical distributed split       |
| Query routing | DB handles automatically | Application/router decides shard |
| Complexity    | Lower                    | Higher                           |

#### 5️⃣ Real System Architecture (Typical)

Application
     │
Shard Router
 ├── Shard1 (Primary + Replica)
 ├── Shard2 (Primary + Replica)
 └── Shard3 (Primary + Replica)

Router decides which shard to query.

if a shard goes down its replica becomes then the primary one

## 1️⃣ Consistent Hashing & Shard Key

🔹 Shard Key

A shard key is a column (or set of columns) used to decide which shard stores a particular record.

It determines data distribution across shards.

e.g: shard key = userId or region

🔹 Consistent Hashing

Consistent hashing is an algorithm used to distribute data across shards such that:

Data is evenly distributed

When a new shard is added or removed, minimal data needs to move

🧠 Critical Example

Suppose we shard a USER table using UserID.

Shard key:
- Shard Key = UserID

Using a hash function:
- Shard = hash(UserID) % 3

| UserID | hash(UserID)%3 | Shard   |
| ------ | -------------- | ------- |
| 101    | 1              | Shard 1 |
| 102    | 2              | Shard 2 |
| 103    | 0              | Shard 0 |

Data gets distributed across shards.

#### 🔹 Problem with Simple Hashing

If we add a new shard:
- hash(UserID) % 4

Almost all data must move to new shards.

Very expensive.

#### 🔹 Solution: Consistent Hashing

Instead of % N, consistent hashing places:
- Servers
- Data
on a hash ring.

Each shard handles a section of the ring.

Key idea

When a shard is added:
- Only a small portion of data moves.

📊 Simple Example

Hash Ring:

0 -------- 100 -------- 200 -------- 300 -------- 360
      |           |            |
    Shard A     Shard B      Shard C

If a new shard is added between B and C:

Only data in that section moves.

#### 2️⃣ Hotspot Problem in Sharding

A hotspot occurs when:
- Too many queries or writes go to a single shard, overloading it while other shards remain underutilized.

Suppose we shard by:
- Shard Key = Region

Data distribution:

| Region | Shard   |
| ------ | ------- |
| US     | Shard 1 |
| EU     | Shard 2 |
| Asia   | Shard 3 |

If 90% users are from US:
- Shard1 → overloaded
- Shard2 → idle
- Shard3 → idle

This creates a hotspot.

#### 🛠 How Hotspots Are Solved

1️⃣ Choose Good Shard Key ⭐

Shard key should be:
- High cardinality
- Evenly distributed
- Frequently used in queries

Example:

UserID → good
Country → risky

2️⃣ Hash-Based Sharding

Use hashing to randomize distribution:

Shard = hash(UserID)

Ensures even load.

3️⃣ Consistent Hashing

Helps rebalance data when shards change.

4️⃣ Virtual Nodes

Each shard has multiple logical nodes to balance load.

Used in large distributed systems.
