## Stateless and Statefull Web Architecture

#### Session Data/State :
data that represents users session on web
- help us in:
    - indentify a user
    - authenticate a user

![alt text](<0.public/image copy 13.png>)

in this diagram: we can see the req from userA must go to server 1 only. so its a overhead
- to solve this we can use sticky sessions inside load-balancer
another problem is that : what if we wanted to remove server-1 / or failure of server-1: then we had to move all the users data to another server:=> overhead

in these cases: Stateless architecture help us

Statefull Web:
- Remebers client data
- stores state info
- overhead, not simple: as seen above inside the diagram

Stateless Web:

![alt text](<0.public/image copy 14.png>)

- does not remember client data
- does not store state information
- simple & robust
- req can go to any server, which then fetches data from shared data storage
