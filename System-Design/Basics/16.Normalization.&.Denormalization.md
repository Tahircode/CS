## Normalization & Denormalization in RDBMS

#### Normalization:
breaking down large tables into smaller, related tables and establishing relationships between them
  - minimizes redundancy & simplification
- do not do over normalization

in simple language:

            Table [ | | | | ]
                 / \
                /    \
        T1 [ | | ]   T2 [ | | ]

#### Denormalization
- deliberatly introducing redundancy into a database table
- opposite of normalization

multiple tables -> merged into single Table

BUT WHY???

denormalization has some benefits
- improve query performance: no join 
- simplified query: ( query on s single table)
- Reduce joins
- Read intensive scanrois : Denormalization is best for read operation scanarios

Cons of denormalization
- redundancy : m/m waste
- Inconsistant Data : (suppose you forgot to update Department location of HR in some rows)
- update/writ operations : extremely slow because you will have to update data in multiple places ( redundant)
